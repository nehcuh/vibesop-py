## 我们的信仰

### 核心信念

**1. 发现 > 执行**

AI 工具已经强大到可以执行大多数任务。真正的问题是：**找到正确的工具**。

- ❌ **错误**: "让 AI 更强大" → 已经够强大了
- ✅ **正确**: "让找工具更容易" → 这是真正的瓶颈

### 2. 匹配 > 猜测

人类的直觉和记忆是有限的：
- 你记不住 45+ 个技能的命令
- 你无法准确判断每个场景该用什么
- 猜测会导致错误的工具选择

- ❌ **错误**: "记住更多命令" → 违反人性
- ✅ **正确**: "智能匹配意图" → 符合认知规律

### 3. 记忆 > 智能

"智能"往往是幻觉。真正的智慧是**记住什么有效**。

- ❌ **错误**: "让 AI 更智能" → AI 已经很聪明了
- ✅ **正确**: "记住用户选择" → 数据驱动的优化

### 4. 开放 > 封闭

没有单一工具可以解决所有问题。真正强大的是**生态**。

- ❌ **错误**: "绑定一个平台" → 被平台锁定
- ✅ **正确**: "跨平台通用" → 工具可替换

## 我们的使命

### 愿景

AI 辅助开发的工具爆炸式增长：
- Claude Code, Cursor, Continue.dev, Aider, Cline...
- 每个工具都有自己的命令和技能
- superpowers, gstack, omx 等技能包蓬勃发展
- 用户不知道该用哪个

### 问题

**认知负荷**: 用户需要记忆 45+ 个技能和命令

**决策困难**: 不知道哪个技能最适合当前场景

**学习曲线**: 每个新工具都要重新学习

**碎片化体验**: 不同工具的命令和技能互不兼容

### 解决方案

**VibeSOP = SkillOS = 智能路由 + 任务编排 + 生命周期管理 + 生态集成**

1. **理解意图** → 自然语言输入
2. **智能匹配** → 8 层路由 pipeline
3. **学习偏好** → 越用越准确
4. **跨平台** → 不绑定任何工具

## 核心原则

### 原则 1: 意图优先

**Don't guess, understand.**

**解读**: 不要让用户猜测命令，要理解用户的真实意图。

**实践**:
- 8 层路由 pipeline，逐层理解
- AI 语义分析（Layer 0）
- 场景模式匹配（Layer 2）
- 多维度相似度计算（Layer 3-6）

### 原则 2: 数据驱动

**Trust data, not intuition.**

**解读**: 相信数据，不依赖直觉。

**实践**:
- InstinctLearner: 学习用户偏好
- MemoryManager: 记住有效选择
- 反馈收集: 持续优化准确率

### 原则 3: 渐进增强

**Start simple, iterate rapidly.**

**解读**: 从简单开始，快速迭代。

**实践**:
- v4.0: 基础路由（85% 准确率）
- v4.1: 优化权重（94% 准确率）
- 未来: 持续优化到 95%+

### 原则 4: 开放生态

**Be open, not closed.**

**解读**: 保持开放，不锁定。

**实践**:
- SKILL.md 标准规范
- 多平台适配器支持
- 外部技能包集成
- 开放的反馈机制

## 设计哲学

### 简单性

**Less is more.**

- **用户端**: 一个命令解决所有问题
- **开发端**: 清晰的 SKILL.md 格式
- **架构**: 三层解耦，易扩展

**避免**:
- 过度的配置
- 复杂的命令行
- 晦能的默认值

### 透明性

**Show your work.**

- **开源代码**: 所有算法可见
- **清晰文档**: 解释决策原因
- **可验证**: 基准测试数据

**实践**:
- 架构决策记录（ADR）
- 优化报告（前后对比）
- 性能基准测试

### 可扩展性

**Design for evolution.**

- **接口化**: SKILL.md 标准化
- **模块化**: 三层架构
- **可插拔**: 匹配器协议

**实践**:
- 新技能: 创建 SKILL.md
- 新平台: 实现适配器
- 新匹配器: 实现 IMatcher

## 核心价值观

**体现**: 智能路由，而非技能执行

### **Matching over Guessing** (匹配 > 猜测)

**含义**: 理解意图比猜测命令更重要

**含义**: 记住有效选择比"更聪明"更重要

**含义**: 开放生态比封闭系统更有价值

**体现**: SKILL.md 标准，跨平台支持

## 我们不做什么

### 我们是什么

| ✅ | 说明 |
|----|------|
| **SkillOS（技能操作系统）** | 技能的全生命周期管理层 |
| 智能路由器 | 理解意图，匹配最佳技能 |
| 任务编排器 | 分解复杂请求，规划执行顺序 |
| 技能生命周期管理器 | 启禁用、作用域隔离、质量评估、自动淘汰 |
| 跨平台适配层 | 一套技能定义，所有 AI Agent 通用 |
| 开放生态 | 任何人可以发布、分享、安装技能 |

### 我们不是什么

| ❌ | 说明 |
|----|------|
| 技能执行器 | 执行技能代码的是 AI Agent，不是 VibeSOP |
| AI 编码工具 | VibeSOP 管理技能，AI Agent 执行技能 |
| 单平台工具 | 不绑定 Claude Code、Cursor 或任何特定平台 |
| 闭源系统 | 永久开源（MIT），开放决策过程 |

> **执行能力说明**: VibeSOP 保留轻量级本地执行能力（`ExternalSkillExecutor`、AST 安全评估器）用于开发者测试和技能验证。但在生产流程中，技能的最终执行由 AI Agent（Claude Code、OpenCode、Cursor 等）完成。这是有意的边界：SkillOS 管"用什么技能、以什么顺序"，AI Agent 管"怎么执行"。

> **定位边界**: VibeSOP 是 SkillOS，管理技能的**全生命周期**：
> **发现 → 安装 → 路由 → 编排 → 执行计划生成 → 评估 → 保留/淘汰**
>
> 其中"执行"环节由 AI Agent（Claude Code、OpenCode、Cursor 等）完成。
> 这是有意的架构分层：SkillOS 管"用什么技能、以什么顺序"，
> AI Agent 管"怎么执行"。两者解耦，确保平台无关性。

### 我们不追求

| ❌ | ✅ |
|----|----|
| 100% 准确率 | 95% 准确率（实用） |
| 0ms 延迟 | <100ms 延迟（够快） |
| 所有技能 | 100+ 主流技能（够用） |
| 所有平台 | 10+ 主流平台（覆盖） |
| 无限扩展 | 有序演进（可持续） |
